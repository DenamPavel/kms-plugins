# AGENTS.md Conventions

This bundled reference defines the conventional structure and quality bar for `AGENTS.md` files generated by the `doc-agents-md` distiller. It is read by the distiller as part of its required reading to ensure sections and guidance match this established convention.

## The Six Conventional Sections

Every project `AGENTS.md` follows this structure:

### 1. Commands

Build commands for the project. Copy the command **verbatim** from the build manifest (e.g., `package.json#scripts.build`, `Makefile`). Cite the source. Example:

```
Commands

Build: npm run build (from package.json#scripts.build)
```

If the investigation could not verify a command, flag it and note the reason.

### 2. Testing

Test commands. Verbatim from the manifest with source. Example:

```
Testing

npm test (from package.json#scripts.test)
```

### 3. Project Structure

High-level components and their responsibilities. Keep it specific and short. Name real modules/services. Example:

```
Project Structure

The system has two main components:
- Event ingester: Validates and writes incoming events to the store (src/ingest.js).
- Rollup reducer: Consumes events, produces time-windowed rollups (src/reduce.js).
```

### 4. Code Style

Project-specific style practices. Distill from the codebase's actual patterns, not generic advice. Example (good):

```
Code Style

Emit rollups by window close time, not by event arrival, to maintain the invariant that no late-arriving event can change a closed rollup.
```

Example (bad—too generic):

```
Code Style

Write clean, readable code. Use descriptive names. Keep functions small.
```

### 5. Git Workflow

Practices for commits, branches, and reviews. If the investigation did not surface explicit practices, insert `[TODO: document team review norms and merge policy]`. Example:

```
Git Workflow

[TODO: document merge approval requirements and release cadence]
```

### 6. Boundaries

What agents or tools may and may not do. Distill the `agentBoundaryBlock` into concrete, project-specific rules. Pair prohibitions with alternatives. Example (good):

```
Boundaries

Always: Read source files, reference module names, cite source paths.
Ask first: Execute build or test commands.
Never: Commit API keys or internal service hostnames to the repository; use environment variables instead.
```

Example (bad):

```
Boundaries

Always: Follow best practices.
Never: Break things.
```

## Quality Bar

Useful `AGENTS.md` files share these characteristics:

- **Non-obvious:** Project-specific guidance, not generic filler. Assume the reader knows how to write clean code; tell them what is special about *this* project.
- **Specific:** Name real components, modules, file paths, and practices. No hand-waving.
- **Concise:** Total file under 150 lines. Respect the reader's time.
- **Pairing:** Every prohibition has an alternative. "Don't X; do Y" instead of "never X."
- **Grounded:** Every fact comes from investigation, never invention. Use `[TODO]` for human judgment, not generic boilerplate.

## Common Mistakes

| Mistake | Fix |
|---------|-----|
| Generic advice ("write clean code") | Replace with project-specific constraint ("emit rollups by window close, not arrival") |
| Prohibition without alternative ("never hardcode secrets") | Pair it ("never hardcode secrets; use environment variables") |
| Invented sections ("remember to be nice to your teammates") | Insert `[TODO]` for human judgment instead |
| Leak instances in Boundaries ("never emit sk-live-abc123") | Phrase generically ("never commit API keys") |
| Filler to reach some minimum length | Cut it. Under 150 lines is fine. |
| Commands copied from prose instead of manifests | Copy verbatim from build/test/run manifests only |
