# docsuite — Test Requirements (Phases 1–5)

Maps each in-scope acceptance criterion to how it is verified. This plugin (`kms-docs` v2.1.0) has **no automated unit-test framework**. Per its conventions (`HANDOFF-screenshots.md`, `STATUS.md`), verification is one of:

- **subagent-RED-GREEN** — dispatch the edited prompt/rulebook to a real subagent with an ad-hoc input; confirm correct behavior (GREEN) and confirm the pre-edit version failed (RED).
- **node-smoke-test** — local Node run of a bundled script, exercising success + failure paths (the `capture.mjs` convention).
- **operational-end-to-end (human-gated)** — a real `/write-doc` run against a fixture app, driven through the two human gates. Human-verified by nature at the gates.
- **static-grep** — a per-phase ad-hoc grep for the standalone-posture check (no external plugin/skill/agent invoked). The *bundled cross-plugin static-check script* itself and AC6.1/6.3/6.4 are Phase 8, out of scope.
- **dispatch-inspection** — give a subagent the edited `SKILL.md` and ask which components it selects for a mode; confirm the correct dispatch-table row/branch. Verifies wiring, not end-to-end execution.

Fixtures in this plan are **ad-hoc, created at verification time**. The durable, curated, hand-labeled fixtures (deliberately-omitted-subsystem app, planted-secret app, hand-labeled must-reject `AGENTS.md` lines) and the formal standalone acceptance test are **Phase 8, out of scope for this plan**.

Scope note: AC2, AC5, AC7, AC6.1, AC6.3, AC6.4 are **deferred to Phases 6–8** and carry no test obligation here. AC6.2 is in scope only as the *per-phase ad-hoc grep* slice (the bundled cross-plugin static-check script is Phase 8).

## Notes: dispatch-wiring-only vs. end-to-end, and ad-hoc-fixture dependence

### Verified only at the "dispatch-wiring" level in Phase 1, completed end-to-end later

Two AC cases are verified in Phase 1 **only** as a dispatch-table selection decision (via subagent inspection of the edited `SKILL.md`), because the agents they name do not exist until later phases. Their end-to-end execution is completed later:

- **AC1.2** (`mode=maintainer`) — wiring verified Phase 1 Task 5; end-to-end verified **Phase 4 Task 4** once `doc-internals-investigator`/`doc-internals-critic` (Phase 3) and `writing-internals` (Phase 2) exist.
- **AC1.6** (`mode=agents-md`) — wiring verified Phase 1 Task 5; end-to-end verified **Phase 5 Task 6** once `doc-agents-md` (Phase 5) exists.

Phase 1's own executable claim is limited to: `user-guide` is behavior-preserving (AC1.1), unknown modes are refused (AC1.3, fully), and the mode plumbing selects the correct row/branch (AC1.2/AC1.6 at the wiring level).

### AC cases dependent on ad-hoc fixtures built at verification time

Every operational and subagent check in this plan uses fixtures **created ad hoc at verification time**, not curated durable fixtures. The following AC cases each depend on a specific ad-hoc fixture being *correct*, not merely present:

- **AC3.1, AC3.5** — a minimal fixture app (ingest → reduce, `package.json` scripts, one invariant, a **planted secret + internal hostname**). Built in Phase 3 Task 3, extended in Phase 4 Task 4.
- **AC3.2** — the same fixture plus an ad-hoc maintainer draft that **deliberately omits the `reduce` subsystem**. Phase 3 Task 5.
- **AC3.3, AC3.4** — ad-hoc machinery paragraphs and a draft with the planted secret/hostname pasted in, plus a reversed data-flow claim. Phase 2 Task 2, Phase 4 Task 4.
- **AC4.2** — ad-hoc `AGENTS.md` + `manifest-values.json` inputs to the Node smoke test. Phase 5 Task 2.
- **AC4.1, AC4.3, AC4.4, AC4.5, AC4.7** — a small fixture repo with real `build`/`test` scripts, a planted secret + hostname, and **hand-labeled must-reject generic lines**. Phase 5 Task 6.

**Phase 8 (out of scope for this plan) later builds durable, curated versions** of these fixtures — the deliberately-omitted-subsystem app, the planted-secret app, and the repo with hand-labeled must-reject `AGENTS.md` lines — so the ACs run against stable, reviewed fixtures rather than fixtures reconstructed at each verification. The Phase 1–5 verifications are complete and self-contained on their ad-hoc fixtures and do not depend on Phase 8.

### Human-gated AC cases

The two human gates (GATE 1 scope, GATE 2 result / the `agents-md` single result gate) are human-verified by nature. AC cases that pass through a human gate as their completing verification: **AC3.3** (GATE 1 + GATE 2), **AC4.1, AC4.3, AC4.4, AC4.5, AC4.7** (result gate; AC4.5 also GATE 1). **AC3.1** (doc half) and **AC3.4** (full pipeline run) also complete inside a human-gated run in Phase 4, though their core claims are also independently checked by subagent/schema checks.
