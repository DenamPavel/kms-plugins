# docsuite — Test Requirements (Phases 6–8)

Maps each in-scope acceptance criterion to how it is verified. This plugin (`kms-docs`) has **no automated unit-test framework**, and that is by design: the components are prompt-driven markdown agents, skills, and commands, plus two Node scripts (`capture.mjs`, `agents-md-filter.mjs`) and one new shell script (`standalone-check.sh`). There is no test runner to add tests to, and inventing one would not fit the repo. The Phases 1–5 `test-requirements.md` set the precedent this file continues: verification is behavioral (dispatch an agent or run a skill against a fixture, inspect the output against the AC) plus a few deterministic script and grep checks.

## Verification types

Every AC below is tagged with one of three types.

- **deterministic-automated** — a check with a fixed pass/fail oracle that a machine decides: the `standalone-check.sh` positive and negative runs, the `agents-md-filter.mjs` mechanical filter, the version-sync JSON parse, the `node_modules`-absence `find`, the `capture.mjs` guard-string grep, and marker/`grep` assertions over file contents. No human judgment.
- **behavioral-fixture** — dispatch a real subagent (or run the orchestrator skill) against a fixture, then judge the structured output against the AC and, where one exists, against the fixture's `LABELS.md` oracle. The judgment is made by the running agent or a human reviewer. Examples: surveyor proposals, the 6-gate count, ledger state transitions, resume behavior, failure-policy branches, the leak producer/consumer tests, the judgment filter, first-run detection, audit-vs-write behavior, the clean-profile standalone run.
- **human-verification** — inherently needs a person, and cannot be fully automated. Two kinds appear here: the human gates themselves (the set gate and each doc's GATE 1/GATE 2/result gate, which are the point of the design and are human by construction), and the outward, hard-to-reverse actions (the gitlab `origin` push and the `master` publish), which the plan explicitly holds for operator confirmation.

A behavioral-fixture verification that completes *inside* a human-gated run carries both tags: its core claim is checkable by an agent, but the run reaches it only by passing a human gate.

## Fixture strategy

Phases 6 and 7 verify against **ad-hoc fixtures created at verification time** — the same approach Phases 1–5 used (see `docs/implementation-plans/2026-07-01-docsuite/test-requirements.md:11`). The survey and orchestration checks build a throwaway app or git repo in a `mktemp` dir, dispatch against it, and inspect the result. These fixtures are named "ad-hoc at verification time" in the tables below.

Phase 8 builds the **durable, curated, hand-labeled fixtures** under repo-level `tests/docsuite-fixtures/` (Task 3), each with a `LABELS.md` that is the test oracle:

- `tests/docsuite-fixtures/fixture-webapp/` — runnable app with a safe/demo mode, a data-flow across ≥2 modules, and a planted secret + token + internal hostname. `LABELS.md` lists the planted-secret strings that must never appear in any emitted artifact. Serves the capture path (AC6.1, AC6.3) and the leak producer/consumer re-verifications (AC3.5, AC4.7).
- `tests/docsuite-fixtures/fixture-omitted-subsystem/` — app with a subsystem deliberately left undocumented relative to an included draft. `LABELS.md` names the subsystem the critic must flag (AC3.2).
- `tests/docsuite-fixtures/fixture-agents-md-labels/` — repo with real manifest scripts plus a `LABELS.md` of hand-labeled must-reject `AGENTS.md` lines: manifest-equal lines for the mechanical filter (AC4.2) and generic advice for the judgment filter (AC4.3).

Phase 8's re-verifications (Task 4) re-run the already-shipped Phase 3–5 agents against these durable fixtures, because "each AC depends on the fixture being correct, not merely present." That is why AC3.2, AC3.5, AC4.2, AC4.3, and AC4.7 appear here even though Phases 3–5 own them: Phase 8 hardens them against reviewed oracles rather than fixtures reconstructed at each run.

## Implementation decisions this file is rationalized against

- **Investigation success = schema-valid `grounding.json` present.** The failure policy (AC5.4) keys on a checkable artifact, not narrated status: the maintainer investigation succeeded only if `$SCRATCH/grounding.json` exists and passes the pinned schema-validity check. Absent or invalid → `investigation-failed` → `AGENTS.md` halted.
- **The standalone check excludes declaration sentences and `kms-docs` self-refs.** `standalone-check.sh` greps `ed3d-|general-purpose|kms-[a-z]` over the invoked pieces, strips `kms-docs` before re-testing, and allows the standalone-posture declaration wording via its `ALLOW` pattern. So a sentence saying "invokes no `ed3d-*` agent" does not trip it, but a real `ed3d-*` or `kms-*`-sibling invocation does.
- **First-run detection lives in Stage 1.5.** Detection is placed inside the capture stage, which already runs only in `user-guide` mode with a non-empty capture plan. It is therefore mode-aware by construction and cannot fire for `maintainer`/`agents-md`.
- **The grounding artifact lives outside the target repo.** It is written as `grounding.json` under a `mktemp -d` scratch dir the orchestrator owns, never at `<repo>/docs/.kms-run/` (a stale, superseded path). The ledger is the only committed run artifact, at `<repo>/docs/.docsuite-ledger.md`.
- **Set-gate corrections are authoritative.** Corrections are written into `$SCRATCH/set-gate-facts.md` and override investigation-derived facts on conflict downstream (visible in `AGENTS.md`).

### ACs that cannot be fully automated, and why

- **AC2.5 / AC5.1 / AC5.5** — depend on a person acting at the **set gate** (reject; count; enter a correction). The consequences are deterministically checkable, but reaching the AC-defining moment requires a human at a human gate. The gates are the design's chosen quality/control mechanism, not an incidental cost.
- **AC4.3 / AC4.7** — complete inside a run backed by the **result gate**; AC4.3 is additionally a model-judgment oracle rather than a mechanical check.
- **AC6.1** — the **clean-profile end-to-end run** must be driven and observed by a person (and passes through the human gates). The static check and agent enumeration are automated proxies, but the plan requires the clean-profile run as the only true end-to-end standalone proof.
- **AC6.4** — the **gitlab `origin` push and `master` publish** are external, hard-to-reverse actions held for explicit operator confirmation. Every other part of AC6.4 (version sync, `node_modules` absence, validation) is deterministic.

All remaining ACs are either deterministic-automated or behavioral checks whose AC-defining signal is objective (artifact presence, schema validity, ledger-state decision, `LABELS.md` match, grep/JSON assertion) and do not require a person at the deciding moment.
