## Definition of Done

`kms-docs` expands from a single-page writer into a multi-document project documentation pipeline.

**What gets built:**

1. **Survey/triage stage.** Given an app repo, propose the doc set that project should have (default: user guide + maintainer/architecture doc + `AGENTS.md`), reconciled against docs that already exist, and present it at a new **set-level human gate**.
2. **Maintainer/architecture doc capability.** A new machinery-is-the-subject writing rulebook (inverted leak model: secrets/real-data only), a re-aimed investigator target, and an internals-flavored coverage critic. Reuses the existing fact-checker, editor, reviser, and two-gate pipeline.
3. **`AGENTS.md` generation capability owned by the plugin** — so a coworker's agents get up to speed with no ed3d tooling. `AGENTS.md` is the cross-tool standard; human docs still only backlink to it. Do not rebuild `CLAUDE.md` machinery.
4. **Higher-level orchestration entry point.** Runs survey → set gate → per-doc pipeline, where each doc keeps its existing two gates (scope + final). Tracks which docs are done.

**How we'll know it's done:**

- A coworker with *only* `kms-docs` installed points it at an app repo, approves the proposed set, and gets all three doc types, each through the two-gate review.
- The maintainer doc documents machinery (not surface) and passes an internals coverage critic; the user-guide path is unchanged.
- A generated `AGENTS.md` gets a fresh agent up to speed, with no ed3d dependency.
- Standalone integrity is audited and preserved, including documented first-run setup for the screenshot path.

**Out of scope for v1** (designed so a pivot stays possible):

- Documenting non-app *processes* (apps first).
- Ops/runbook and internal-API/reference doc types.

**Gate model (resolved trade-off):** one set-level survey gate + the existing two gates per doc. Chose quality/control over maximal turnkey automation.

## Acceptance Criteria

### docsuite.AC1: The per-doc engine is mode-parameterized and backward-compatible
- **docsuite.AC1.1 Success:** A `/write-doc` invocation with no mode dispatches `doc-investigator`, `doc-coverage-critic`, and `writing-documentation` — identical to v2.1.0 behavior.
- **docsuite.AC1.2 Success:** A `/write-doc` invocation with `mode=maintainer` dispatches `doc-internals-investigator`, `doc-internals-critic`, and `writing-internals`.
- **docsuite.AC1.3 Failure:** An unrecognized mode value is rejected with a clear error, not silently treated as `user-guide`.
- **docsuite.AC1.4 Edge:** The shared prose/voice rules exist in exactly one file; both `writing-documentation` and `writing-internals` reference it (no duplicated tells checklist).

### docsuite.AC2: The survey proposes and reconciles a doc set
- **docsuite.AC2.1 Success:** For an app repo, `doc-surveyor` emits a proposed doc set (default taxonomy) plus an extracted-facts bundle (build/test/run commands, structure, stack).
- **docsuite.AC2.2 Success:** For a non-user-facing tooling repo, the survey drops the user guide from the proposal.
- **docsuite.AC2.3 Success:** Each existing doc is marked write-fresh vs audit-existing.
- **docsuite.AC2.4 Failure:** An app not runnable in a safe capture mode is flagged (screenshots may be skipped), surfaced at the set gate, not discovered mid-run.

### docsuite.AC3: The maintainer doc documents machinery
- **docsuite.AC3.1 Success:** `mode=maintainer` produces a doc describing data flow, module boundaries, invariants, and why-decisions for a fixture app, and the internals grounding artifact is schema-valid.
- **docsuite.AC3.2 Success:** `doc-internals-critic` flags a deliberately omitted subsystem in a fixture.
- **docsuite.AC3.3 Success:** The maintainer doc clears GATE 1 and GATE 2.
- **docsuite.AC3.4 Failure:** A maintainer doc that leaks a secret/token/internal hostname/real-production-datum is caught (leak model), while architecture/module/schema-shape content is permitted.

### docsuite.AC4: AGENTS.md is a non-generic, gated distillation
- **docsuite.AC4.1 Success:** Generated `AGENTS.md` contains copy-pasteable build/test commands and the ✅/⚠️/🚫 boundaries distilled from investigation.
- **docsuite.AC4.2 Failure:** Lines derivable from `cat package.json`/`Makefile`, or advice true of every repo, are filtered out (signal-density disqualifiers).
- **docsuite.AC4.3 Success:** Human-judgment sections appear as `[TODO]` placeholders; the file passes through its result gate before being written.
- **docsuite.AC4.4 Success:** Requesting `AGENTS.md` without a maintainer doc runs a full internals investigation (not a shallow variant) + its own GATE 1 + result gate.
- **docsuite.AC4.5 Edge:** The optional `CLAUDE.md` bridge uses the syntax Claude Code actually supports (verified), and is offered, not forced.

### docsuite.AC5: Orchestration runs the gates, tracks, and resumes
- **docsuite.AC5.1 Success:** A default 3-doc run hits exactly 6 gates (1 set + 2 + 2 + 1).
- **docsuite.AC5.2 Success:** The ledger in `<repo>/docs/` records per-doc status through the defined states.
- **docsuite.AC5.3 Success:** Re-invoking `/document-project` skips docs already marked `done`.
- **docsuite.AC5.4 Failure:** A failed maintainer investigation blocks the dependent `AGENTS.md`; independent docs may still proceed.
- **docsuite.AC5.5 Edge:** Corrections made at the set gate propagate into the facts bundle `AGENTS.md` later consumes.

### docsuite.AC6: Standalone integrity and first-run setup
- **docsuite.AC6.1 Success:** The full pipeline runs with only `kms-docs` installed (no ed3d, no `kms-human-voice`).
- **docsuite.AC6.2 Failure:** A static check finds no invocation of any external plugin, skill, or general-purpose agent from the new pieces.
- **docsuite.AC6.3 Success:** First-run `npm install` detection fires only when a capture stage is in scope, and prints the exact command; it does not fire for `maintainer`/`agents-md`.
- **docsuite.AC6.4 Success:** `kms-docs` and the marketplace are version-bumped and the gitlab export is synced.

### docsuite.AC7: Cross-cutting — ask, don't guess
- **docsuite.AC7.1 Success:** The surveyor asks the runner which data/accounts/verticals/fixtures are safe to capture and whether a safe mode exists, recording answers into the capture plan.
- **docsuite.AC7.2 Failure:** `capture.mjs` still refuses any shot whose safe-mode assertion is absent (code-enforced backstop retained).

## Architecture

Two orchestration layers over one shared investigation.

### Two layers

**Per-doc engine (mode-parameterized).** The existing `documentation-pipeline` skill and `/write-doc` command remain the engine that produces one doc through `ground → GATE 1 (scope) → capture → draft → cross-model review → revise → GATE 2 (result)`. This design adds a **doc-type mode** (`user-guide` | `maintainer` | `agents-md`) that selects which investigator, coverage critic, and writing rulebook the engine dispatches. This is a real modification to the shipped engine, not a passive default: `documentation-pipeline/SKILL.md`, `commands/write-doc.md`, and several agent prompts hardcode agent names and the `writing-documentation` reference today, and mode threading replaces those literals with a dispatch table. The `user-guide` mode reproduces current behavior exactly, so a bare `/write-doc` invocation is behavior-preserving.

**Project layer (new).** A `documenting-a-project` skill and `/document-project <repo-path>` command orchestrate a whole project: run the survey, hold the set gate, invoke the per-doc engine once per doc in the right mode, and maintain a ledger. It is pure orchestration and dispatches only bundled agents.

### The mental model that resolves the doc dependencies

The **internals investigation** is the unit `AGENTS.md` depends on; the **maintainer doc** is an optional second consumer of that same investigation. Concretely:

- `AGENTS.md` alone → run the full `doc-internals-investigator`, GATE 1, distill, result gate.
- `AGENTS.md` + maintainer doc → one investigation feeds both.
- maintainer doc alone → investigation, full doc pipeline, its two gates.

There is exactly one internals-investigation behavior (no "shallow mode" variant). Doc production is what varies, not grounding depth.

### Mode dispatch

The engine selects components by mode:

| Mode | Investigator | Coverage critic | Writing rulebook |
|------|-------------|-----------------|------------------|
| `user-guide` (default) | `doc-investigator` | `doc-coverage-critic` | `writing-documentation` |
| `maintainer` | `doc-internals-investigator` | `doc-internals-critic` | `writing-internals` |
| `agents-md` | `doc-internals-investigator` | (none; signal-density check at gate) | (distiller, not prose) |

`doc-fact-checker`, `doc-editor`, `doc-reviser`, `doc-screenshooter` are mode-agnostic and shared as-is. Mode reaches the engine as an explicit argument the orchestrator passes on dispatch; `/write-doc` gains an optional trailing mode argument defaulting to `user-guide`.

### Data flow (default 3-doc run)

```
/document-project <repo>
  → doc-surveyor: classify repo, inventory existing docs, extract facts bundle,
                  ask safe-capture questions
  → SET GATE (approve doc set + per-doc write-vs-audit + scope)
  → per doc, sequenced:
      user guide (mode=user-guide): full engine, GATE 1 + GATE 2
      maintainer  (mode=maintainer): doc-internals-investigator writes the
                  internals grounding artifact; GATE 1 + GATE 2
      AGENTS.md   (mode=agents-md): reuse the internals grounding artifact +
                  facts bundle; doc-agents-md distills; single result gate
  → update ledger in <repo>/docs/
```

### The internals grounding artifact (contract)

The internals investigation writes a **run-scoped artifact** so the maintainer writer and the `AGENTS.md` distiller both consume one investigation across separate engine invocations.

- **Path:** `<target-repo>/docs/.kms-run/internals-grounding.md` (the `.kms-run/` directory is gitignored; the orchestrator creates it and removes it at run end).
- **Schema (pinned):** machinery map (data flow, module boundaries), invariants, why-decisions, extracted build/test/run facts, and an explicit **agent-boundary block** structured as ✅ always / ⚠️ ask-first / 🚫 never — the exact structure `doc-agents-md` distills from.
- **Lifetime:** run-scoped, not committed. The durable, committed record is each doc's **page-to-source map**, so a later audit re-grounds against current source, never against a stale artifact.

### Ask, don't guess (pipeline norm)

Agents stop and ask the runner rather than guess whenever a decision affects leak risk or ground truth is genuinely ambiguous. "What is safe to screencap" is an explicit unknown the pipeline must ask about, never infer. This lands as concrete instructions in `doc-surveyor` (asks which data/accounts/verticals/fixtures are safe and whether a safe mode exists) and the mode agents, not as a slogan. `capture.mjs`'s code-enforced safe-mode assertion remains the backstop.

## Existing Patterns

Investigation of `plugins/kms-docs` (v2.1.0) established the patterns this design follows:

- **One-purpose bundled agents** in `agents/*.md`, each with a pinned model and a fixed toolset. New agents (`doc-surveyor`, `doc-internals-investigator`, `doc-internals-critic`, `doc-agents-md`) follow this shape and pin models.
- **Orchestrator-skill pattern:** `documentation-pipeline/SKILL.md` dispatches agents and holds gates. `documenting-a-project` is a second orchestrator that composes the first, matching the existing shape rather than inventing a new one.
- **Standalone posture (verified clean):** no ed3d dependency, no *required* `kms-human-voice` (its rules are restated inline in `writing-documentation`), no general-purpose-agent fallback (`documentation-pipeline/SKILL.md` states this explicitly). New pieces preserve this.
- **Scratch-dir + promote workflow:** drafts and screenshots live in a working dir and are promoted on approval; the grounding artifact follows the same run-scoped-then-discarded discipline.
- **Page-to-source map** as the durable doc↔code link, and a **code-to-docs backlink note** in the nearest `AGENTS.md`/`CLAUDE.md`. Retained; the plugin still only *appends a backlink* to those files and does not otherwise author `CLAUDE.md`.
- **Describe-and-link, never duplicate** (`writing-documentation` line 24). This design applies it to skills too: the shared prose/voice rules move to one in-plugin fragment both writing rulebooks reference, rather than duplicating ~130 lines across `writing-documentation` and `writing-internals`.

Divergence: the per-doc engine gains mode-branching it did not have. Justified because the alternative (a separate parallel pipeline per doc type) duplicates the review machinery and the two-gate quality story.

## Implementation Phases

### Phase 1: Mode-parameterize the per-doc engine
**Goal:** The engine accepts a doc-type mode; `user-guide` default is behavior-preserving.

**Components:**
- `plugins/kms-docs/skills/documentation-pipeline/SKILL.md` — add the mode-dispatch table; replace hardcoded `doc-investigator` / `doc-coverage-critic` / `writing-documentation` literals with "the mode's investigator / critic / rulebook."
- `plugins/kms-docs/commands/write-doc.md` — optional trailing mode argument, default `user-guide`.
- `plugins/kms-docs/agents/doc-writer.md` — read the rulebook the mode selects rather than `writing-documentation` by name.
- New shared fragment `plugins/kms-docs/skills/writing-documentation/prose-voice-rules.md` (or a shared skill file) — extract the tells/voice rules; `writing-documentation` references it.

**Dependencies:** None (first phase).

**Done when:** A bare `/write-doc` run dispatches the same agents/rulebook as v2.1.0 (tests verify `docsuite.AC1.1`); the engine dispatches by mode when a mode is supplied (`docsuite.AC1.2`).

### Phase 2: `writing-internals` rulebook
**Goal:** A machinery-is-the-subject writing rulebook with an explicit inverted leak model.

**Components:**
- `plugins/kms-docs/skills/writing-internals/SKILL.md` — scope is machinery (data flow, module boundaries, invariants, why-decisions). References the shared prose-voice fragment from Phase 1. Ambiguous cases (dependency version pins, internal service names) resolved explicitly in the skill.

**Dependencies:** Phase 1 (shared fragment).

**Done when:** The rulebook exists and is self-contained (no external-plugin dependency) — verified by `docsuite.AC3.4` and `docsuite.AC6.2`.

### Phase 3: Internals investigation and critic agents
**Goal:** Agents that ground machinery and audit machinery coverage.

**Components:**
- `plugins/kms-docs/agents/doc-internals-investigator.md` — maps machinery; **emits the pinned grounding-artifact schema** including the ✅/⚠️/🚫 agent-boundary block; writes to `<repo>/docs/.kms-run/internals-grounding.md`. Model pinned (sonnet).
- `plugins/kms-docs/agents/doc-internals-critic.md` — finds undocumented subsystems / data flows / invariants (the machinery analogue of `doc-coverage-critic`). Model pinned (sonnet).

**Dependencies:** Phase 2 (rulebook defines what "documented machinery" means).

**Done when:** The investigator produces a schema-valid artifact (`docsuite.AC3.1`); the critic flags a known-omitted subsystem in a fixture (`docsuite.AC3.2`).

### Phase 4: Maintainer doc mode end-to-end
**Goal:** `mode=maintainer` produces a machinery doc through the full two-gate engine.

**Components:**
- Wire `maintainer` mode through `documentation-pipeline` (investigator, critic, rulebook from the dispatch table); confirm `doc-fact-checker` / `doc-editor` / `doc-reviser` operate unchanged on machinery content.
- Grounding-artifact lifecycle: orchestrator creates `<repo>/docs/.kms-run/`, ensures it is gitignored, removes it at run end.

**Dependencies:** Phases 1–3.

**Done when:** A maintainer doc is produced for a fixture app, documents machinery (not surface), passes the internals critic, and clears GATE 1 + GATE 2 (`docsuite.AC3.1`–`docsuite.AC3.3`).
<!-- END_PHASE_4 -->

<!-- START_PHASE_5 -->
### Phase 5: `AGENTS.md` distillation flow
**Goal:** A gated `AGENTS.md` distilled from internals grounding + extracted facts.

**Components:**
- `plugins/kms-docs/agents/doc-agents-md.md` — consumes the grounding artifact + facts bundle; assembles conventional sections with copy-pasteable commands; distills the ✅/⚠️/🚫 boundaries; **filters by machine-checkable signal-density disqualifiers** (reject any line derivable from `cat package.json`/`Makefile`; reject advice true of every repo); inserts `[TODO]` placeholders for human judgment. Model pinned (opus, for the judgment).
- Single result gate: drafted file, extracted commands (unverified ones flagged), `[TODO]`s, and the optional `CLAUDE.md` bridge.
- `CLAUDE.md` bridge: offer (not force) a thin `CLAUDE.md` that imports `AGENTS.md`. **Exact import syntax verified against Claude Code before shipping** (the `@path` form, not `@import`).

**Dependencies:** Phase 3 (grounding artifact + schema). Runs after the maintainer doc when both are in the set; runs after a standalone internals investigation when alone.

**Done when:** `AGENTS.md` is generated with non-obvious content and no generic filler (`docsuite.AC4.1`–`docsuite.AC4.3`); the standalone (no-maintainer-doc) path works via full investigation (`docsuite.AC4.4`).

<!-- START_PHASE_6 -->
### Phase 6: Survey stage
**Goal:** Propose and reconcile a project's doc set; gather safe-capture answers.

**Components:**
- `plugins/kms-docs/agents/doc-surveyor.md` — classify the repo (manifests, entry points, structure); inventory existing docs (fresh-write vs audit-existing per doc); propose the doc set from the default taxonomy, adjusted for the repo; produce the extracted-facts bundle; **ask the safe-capture questions** and flag apps not runnable in a safe mode. Model pinned (sonnet).

**Dependencies:** None functionally, but sequenced before orchestration (Phase 7 consumes its output).

**Done when:** The surveyor emits a set proposal + facts bundle for a fixture, drops the user guide for a non-user-facing repo, and marks each existing doc write-vs-audit (`docsuite.AC2.1`–`docsuite.AC2.3`).
<!-- END_PHASE_6 -->

<!-- START_PHASE_7 -->
### Phase 7: Project orchestration
**Goal:** `/document-project` runs survey → set gate → per-doc engine, with tracking and resume.

**Components:**
- `plugins/kms-docs/skills/documenting-a-project/SKILL.md` and `plugins/kms-docs/commands/document-project.md`.
- **Set gate:** approve/adjust the doc set, per-doc write-vs-audit, and scope. Corrections here propagate into the facts bundle `AGENTS.md` later consumes.
- **Ledger:** `<repo>/docs/.docsuite-ledger.md` (or `.json`) with per-doc status (`pending | grounded | gate1-passed | drafted | gate2-passed | done | failed`). Read on entry; **skip `done` docs on re-invocation** (resume).
- **Failure policy:** a failed/abandoned doc halts dependent docs; specifically a failed maintainer investigation blocks the `AGENTS.md` that depends on its grounding. Independent docs may continue.
- **Sequencing:** survey → user guide → maintainer → `AGENTS.md`.
- **Audit vs write** passed to the engine as explicit execution intent (audit re-grounds and revises an existing page; write starts fresh).

**Dependencies:** Phases 1, 4, 5, 6.

**Done when:** A full 3-doc run hits exactly the 6 gates, tracks status in the ledger, resumes without re-running `done` docs, and blocks `AGENTS.md` when the maintainer doc fails (`docsuite.AC5.1`–`docsuite.AC5.5`).
<!-- END_PHASE_7 -->

<!-- START_PHASE_8 -->
### Phase 8: Standalone integrity, first-run setup, and release
**Goal:** Ship it, provably standalone, with self-announcing setup.

**Components:**
- **Standalone acceptance test:** run the full pipeline with only `kms-docs` present (no ed3d, no `kms-human-voice`); assert no external plugin/skill/agent is invoked.
- **Mode-aware first-run detection:** when the capture stage is in scope (`user-guide` with screenshots) and `scripts/node_modules` is absent, stop and print the exact `npm install` command, or offer to proceed image-less. Detection does not fire for `maintainer`/`agents-md` (no capture).

**Dependencies:** Phases 1–7.

**Done when:** The standalone test passes (`docsuite.AC6.1`–`docsuite.AC6.2`); first-run detection is mode-aware (`docsuite.AC6.3`); the release is published and the export synced (`docsuite.AC6.4`).

## Additional Considerations

**Extensibility (the deferred `process`/`runbook` pivot).** The mode-dispatch table (Phase 1) and the surveyor's taxonomy (Phase 6) are the two extension points. Adding a future `runbook` or `process` mode means one row in the dispatch table + one taxonomy entry + its agents/rulebook — no change to the orchestration or gate machinery. Noted so the pivot stays cheap; not built in v1.

**Skill count / phase scope.** Eight phases, at the writing-plans limit. If phases split during implementation planning, Phases 1–5 (the engine + maintainer + `AGENTS.md`) form a coherent first implementation plan and Phases 6–8 (survey + orchestration + release) a second.

**AGENTS.md quality risk.** Research showed naive auto-generated `AGENTS.md` files underperform. The mitigation is structural: content comes only from real investigation (never a template), generic/derivable lines are filtered by machine-checkable rules, and the file is presented at its gate as human-curated scaffolding, not a finished artifact.
