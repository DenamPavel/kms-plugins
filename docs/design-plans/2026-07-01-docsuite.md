# docsuite — Project Documentation Pipeline Design

## Summary

`kms-docs` today runs one pipeline for one doc type. This design adds two more doc types (a maintainer/architecture doc and `AGENTS.md`) and a survey stage that proposes a doc set for a whole project, without forking the pipeline into parallel copies. It does this with two orchestration layers: the existing `documentation-pipeline` engine, which still owns the ground → gate → draft → review → revise → gate sequence for one document, and a new `documenting-a-project` layer on top that runs a survey, holds a set-level gate, and calls the per-doc engine once per document. The per-doc engine becomes mode-parameterized — `user-guide`, `maintainer`, or `agents-md` — so a single dispatch table, rather than three separate pipelines, selects which investigator, coverage critic, scope rulebook, and leak model each run uses. The `user-guide` mode is wired to reproduce exactly what the engine does today, so existing single-page behavior doesn't change.

The design's central move is treating the internals investigation as one artifact with two possible consumers. A maintainer doc and `AGENTS.md` both need to understand a project's machinery — module boundaries, data flow, invariants, what's safe for an agent to touch — so rather than run that investigation twice, one investigation writes a schema-pinned grounding artifact to a scratch directory, and the maintainer doc and the `AGENTS.md` distiller each read from it within a single orchestrator run. This is also where leak-safety logic has to flip: a user-guide leak list is about not exposing implementation detail, while a maintainer doc's leak list is inverted (architecture is the permitted subject; only secrets, tokens, and real production data are sensitive), so every leak-aware agent (writer, editor, reviser, fact-checker, and the `AGENTS.md` distiller) has to read its leak model and list from the active mode rather than have that hardcoded.

## Definition of Done

`kms-docs` expands from a single-page writer into a multi-document project documentation pipeline.

**What gets built:**

1. **Survey/triage stage.** Given an app repo, propose the doc set that project should have (default: user guide + maintainer/architecture doc + `AGENTS.md`), reconciled against docs that already exist, and present it at a new **set-level human gate**.
2. **Maintainer/architecture doc capability.** A new machinery-is-the-subject scope rulebook (inverted leak model: secrets/real-data only), a re-aimed investigator target, and an internals-flavored coverage critic. Reuses the existing two-gate pipeline; the writer, editor, and reviser are re-pointed at the machinery scope rulebook, and the fact-checker's leak check reads the mode's leak model and list, so a machinery doc is not judged against user-guide scope.
3. **`AGENTS.md` generation capability owned by the plugin** — so a coworker's agents get up to speed with no ed3d tooling. `AGENTS.md` is the cross-tool standard; human docs still only backlink to it. Do not rebuild `CLAUDE.md` machinery.
4. **Higher-level orchestration entry point.** Runs survey → set gate → per-doc pipeline, where each doc keeps its existing two gates (scope + final). Tracks which docs are done, and owns that ledger exclusively so a direct single-doc run cannot drift it.

**How we'll know it's done:**

- A coworker with *only* `kms-docs` installed points it at an app repo, approves the proposed set, and gets all three doc types, each through the two-gate review.
- The maintainer doc documents machinery (not surface) and passes an internals coverage critic; the user-guide path is unchanged.
- A generated `AGENTS.md` gets a fresh agent up to speed, with no ed3d dependency.
- Standalone integrity is audited and preserved, including documented first-run setup for the screenshot path.

**Out of scope for v1** (designed so a pivot stays possible):

- Documenting non-app *processes* (apps first).
- Ops/runbook and internal-API/reference doc types.
- A lightweight `AGENTS.md` path: `AGENTS.md`-alone still pays for a full internals investigation (see the mental model below), and a cheaper variant is deferred.
- Monorepos with distinct per-package agent boundaries (the surveyor flags them; per-package boundary blocks are deferred).

**Gate model (resolved trade-off):** one set-level survey gate + the existing two gates per doc. Chose quality/control over maximal turnkey automation.

## Acceptance Criteria

### docsuite.AC1: The per-doc engine is mode-parameterized and backward-compatible
- **docsuite.AC1.1 Success:** A `/write-doc` invocation with no mode dispatches `doc-investigator`, `doc-coverage-critic`, `doc-writer`, `doc-editor`, and `doc-reviser` against the user-guide scope rulebook — identical dispatch to v2.1.0.
- **docsuite.AC1.2 Success:** A `/write-doc` invocation with `mode=maintainer` dispatches `doc-internals-investigator`, `doc-internals-critic`, points `doc-writer`, `doc-editor`, and `doc-reviser` at the internals scope rulebook, and feeds `doc-fact-checker` the mode's leak model plus the investigation's inverted leak list.
- **docsuite.AC1.3 Failure:** An unrecognized mode value is refused with a clear error by the command and the orchestrator, not silently treated as `user-guide`. Enforced at the instruction layer; a bad mode is a wiring error, not a leak risk, so it needs no code backstop the way the capture safe-mode assertion does.
- **docsuite.AC1.4 Edge:** The shared prose-voice tells/rules live in exactly one fragment file; `writing-documentation` (user-guide scope) and `writing-internals` (machinery scope) both reference it. Scope rules are mode-specific and are not shared.
- **docsuite.AC1.5 Success:** After the shared prose-voice fragment is extracted, a bare `/write-doc` run produces a draft whose reviewers apply the same tells checklist as v2.1.0. Each of `doc-writer`, `doc-editor`, `doc-reviser` names both its scope rulebook and the shared fragment in its own required-reading, so delivery does not depend on transitive reference-following. This is verified against the installed Claude Code, like the `@path` bridge.
- **docsuite.AC1.6 Success:** A `/write-doc` invocation with `mode=agents-md` dispatches `doc-internals-investigator` with no coverage critic and routes to `doc-agents-md` (the distiller) with no writer/editor/reviser pass.

### docsuite.AC2: The survey proposes and reconciles a doc set
- **docsuite.AC2.1 Success:** For an app repo, `doc-surveyor` emits a proposed doc set (default taxonomy) plus a set-gate facts bundle (build/test/run commands, structure, stack).
- **docsuite.AC2.2 Success:** For a non-user-facing tooling repo, the survey drops the user guide from the proposal.
- **docsuite.AC2.3 Success:** Each existing doc is marked write-fresh vs audit-existing.
- **docsuite.AC2.4 Failure:** An app not runnable in a safe capture mode is flagged (screenshots may be skipped), surfaced at the set gate, not discovered mid-run.
- **docsuite.AC2.5 Edge:** A wholesale rejection at the set gate (operator declines the entire proposed set or cancels) aborts the run cleanly without writing a ledger or any doc.

### docsuite.AC3: The maintainer doc documents machinery
- **docsuite.AC3.1 Success:** `mode=maintainer` produces a doc describing data flow, module boundaries, invariants, and why-decisions for a fixture app, and the internals grounding artifact is schema-valid.
- **docsuite.AC3.2 Success:** `doc-internals-critic` flags a deliberately omitted subsystem in a fixture, checking the draft against the enumerated machinery map in the grounding artifact (a finite, bounded surface), not against "all possible machinery."
- **docsuite.AC3.3 Success:** The maintainer doc clears GATE 1 and GATE 2, with the editor and reviser applying machinery scope (not user-guide scope).
- **docsuite.AC3.4 Failure:** `doc-fact-checker`, reading the investigation's inverted leak model and list, catches a maintainer doc that leaks a secret/token/internal hostname/real-production-datum, while permitting architecture/module/schema-shape content.
- **docsuite.AC3.5 Success:** Given a fixture app with a planted secret/token/internal hostname, `doc-internals-investigator` emits an inverted leak list that contains that instance and does not contain permitted architecture/module-name content. (Producer test, symmetric to AC3.4's consumer test.)

### docsuite.AC4: AGENTS.md is a non-generic, gated distillation
- **docsuite.AC4.1 Success:** Generated `AGENTS.md` contains build/test commands taken verbatim from the investigation's extracted facts and the ✅/⚠️/🚫 boundaries distilled from the investigation.
- **docsuite.AC4.2 Failure (mechanical):** Any line equal to a manifest field value (`package.json` script, `Makefile` target, etc.) after normalizing whitespace and case, or within a small fixed edit distance, is filtered out by a deterministic string check.
- **docsuite.AC4.3 Failure (judgment):** Advice that would hold for an arbitrary repo (generic filler) is rejected by the opus distiller and the result gate. This is a judgment oracle, tested against a fixture with hand-labeled must-reject lines, not a mechanical check.
- **docsuite.AC4.4 Success:** Human-judgment sections appear as `[TODO]` placeholders; the file passes through its result gate before being written.
- **docsuite.AC4.5 Success:** Requesting `AGENTS.md` without a maintainer doc runs a full internals investigation (not a shallow variant) + its own GATE 1 + result gate. Its build/test/run commands come from the investigation's extracted facts, so the standalone path needs no surveyor.
- **docsuite.AC4.6 Edge:** The optional `CLAUDE.md` bridge uses the syntax Claude Code actually supports (the `@path` form, verified against the installed Claude Code), and is offered, not forced.
- **docsuite.AC4.7 Failure:** For a fixture whose investigation surfaced a secret/internal hostname, `doc-agents-md` does not reproduce that instance in the generated `AGENTS.md`, in a distilled section or a ✅/⚠️/🚫 boundary line. (Leak coverage for the committed file, symmetric to AC3.4.)

### docsuite.AC5: Orchestration runs the gates, tracks, and resumes
- **docsuite.AC5.1 Success:** A default 3-doc run hits exactly 6 gates (1 set + 2 + 2 + 1).
- **docsuite.AC5.2 Success:** The committed ledger in `<repo>/docs/` records per-doc status through the defined states.
- **docsuite.AC5.3 Success:** Re-invoking `/document-project` reads the ledger and skips docs already marked `done`; a doc marked `investigation-done` whose scratch grounding artifact is gone is demoted and re-investigated rather than skipped.
- **docsuite.AC5.4 Failure:** A failed maintainer *investigation* blocks the dependent `AGENTS.md`. A maintainer doc that fails at drafting or GATE 2 *after* a valid investigation does not block `AGENTS.md`. Independent docs proceed regardless.
- **docsuite.AC5.5 Edge:** Corrections made at the set gate are written into the facts record the investigation and `AGENTS.md` consume, and override investigation-derived facts on conflict.
- **docsuite.AC5.6 Edge:** Direct `/write-doc` invocations do not read or write the ledger; the ledger is owned solely by `/document-project`, so the two entry points cannot silently drift its state.

### docsuite.AC6: Standalone integrity and first-run setup
- **docsuite.AC6.1 Success:** The full pipeline runs with only `kms-docs` installed (no ed3d, no `kms-human-voice`).
- **docsuite.AC6.2 Failure:** A bundled grep-based static check (written as part of this work, not assumed) finds no invocation of any external plugin, skill, or general-purpose agent from the new pieces.
- **docsuite.AC6.3 Success:** First-run `npm install` detection fires only when a capture stage is in scope, and prints the exact command; it does not fire for `maintainer`/`agents-md`.
- **docsuite.AC6.4 Success:** `kms-docs` and the marketplace are version-bumped, and the gitlab export is synced and confirmed to ship no `node_modules`.

### docsuite.AC7: Cross-cutting — ask, don't guess
- **docsuite.AC7.1 Success:** The surveyor asks the runner which data/accounts/verticals/fixtures are safe to capture and whether a safe mode exists, recording answers into the capture plan.
- **docsuite.AC7.2 Failure:** `capture.mjs` still refuses any shot whose safe-mode assertion is absent (code-enforced backstop retained).

## Glossary

- **Per-doc engine**: The existing `documentation-pipeline` skill and `/write-doc` command — the pipeline that produces a single document through grounding, two human gates, drafting, and review. This design parameterizes it by mode rather than replacing it.
- **Mode / mode-parameterized**: An explicit argument (`user-guide`, `maintainer`, or `agents-md`) that tells the per-doc engine which investigator, coverage critic, scope rulebook, and leak model to dispatch. One engine branches by mode instead of three separate pipelines existing side by side.
- **Scope rulebook**: The skill file (`writing-documentation` or the new `writing-internals`) that tells the writer, editor, and reviser what counts as in-bounds subject matter for a doc — user-facing behavior for `writing-documentation`, machinery (data flow, module boundaries, invariants) for `writing-internals`.
- **Leak model vs. leak list**: Two halves of leak safety. The leak *model* is the category definition of what counts as sensitive (comes from the scope rulebook — e.g., "implementation detail is sensitive" vs. its inverse, "only secrets/tokens/hostnames are sensitive"). The leak *list* is the repo-specific instances an investigator actually found under that model. Agents need both: the list alone can't catch a newly written sentence the investigator never saw.
- **DO-NOT-LEAK list**: The user-guide mode's leak list — the specific implementation details, internal names, and similar instances `doc-investigator` finds and flags as off-limits for a user-facing doc.
- **Agent-boundary block (✅/⚠️/🚫)**: A structured section of the grounding artifact marking which actions a coding agent may always take, should ask before taking, or should never take in that repo. Distinct from the leak list: the leak list governs what text must not appear in any doc, the boundary block governs agent permissions and is what `doc-agents-md` distills into `AGENTS.md`.
- **Grounding artifact**: The schema-pinned file the internals investigator writes to a scratch directory — machinery map, invariants, why-decisions, extracted build/test/run facts, inverted leak list, and boundary block — that both the maintainer doc and `AGENTS.md` read from within one orchestrator run.
- **Page-to-source map**: The durable, committed record linking a doc's claims to the source locations that ground them, so a later audit can re-verify against current code. Unlike the grounding artifact, this persists past the run.
- **Set gate**: The new project-level human checkpoint where the proposed doc set (which docs, write-fresh vs. audit-existing, scope) is approved, adjusted, or rejected before any per-doc work starts.
- **GATE 1 / GATE 2**: The two existing per-doc human checkpoints inside the engine — GATE 1 approves scope and (for user-guide mode) the leak list and capture plan before drafting; GATE 2 approves the final result after review and revision.
- **Ledger**: The committed per-project file (`<repo>/docs/.docsuite-ledger.md`) that `/document-project` uses to track each doc's status across states like `pending`, `investigation-done`, `gate2-passed`, and `done`, enabling resume without redoing finished work.
- **Facts bundle**: The build/test/run commands, repo structure, and stack details the surveyor extracts to inform the set gate proposal — distinct from the more authoritative facts the internals investigator later extracts, which `AGENTS.md` actually consumes.
- **AGENTS.md**: A cross-tool convention (not `kms-docs`- or ed3d-specific) for a file that briefs any coding agent on a repo's build/test commands and action boundaries. This design has the plugin generate one directly, since a coworker without ed3d tooling still needs it.
- **`@path` bridge**: The import syntax Claude Code supports for one file to reference another (used here so an optional thin `CLAUDE.md` can import `AGENTS.md` rather than duplicate it). The design calls for verifying this syntax against the installed Claude Code rather than assuming it.
- **Signal-density filter**: Either of two checks `doc-agents-md` runs before content reaches the result gate — a mechanical filter that strips lines duplicating manifest values, and a judgment filter (opus-driven) that rejects generic advice that would hold for any repo.
- **doc-surveyor**: New agent that classifies a repo, inventories existing docs, proposes the doc set, and produces the set-gate facts bundle.
- **doc-internals-investigator**: New agent, the maintainer/agents-md-mode counterpart to `doc-investigator`, that maps machinery and produces the grounding artifact.
- **doc-internals-critic**: New agent, the maintainer-mode counterpart to `doc-coverage-critic`, that flags undocumented machinery against the grounding artifact's enumerated map.
- **doc-agents-md**: New agent (the "distiller") that consumes the grounding artifact and facts to assemble `AGENTS.md`, applying the two signal-density filters and inserting `[TODO]` placeholders for human judgment.
- **Standalone posture**: The property, carried over from the existing plugin, that the full pipeline runs with only `kms-docs` installed — no ed3d plugin, no `kms-human-voice`, no general-purpose-agent fallback. This design's new pieces are required to preserve it, verified by a bundled static check.

## Architecture

Two orchestration layers over one shared investigation.

### Two layers

**Per-doc engine (mode-parameterized).** The existing `documentation-pipeline` skill and `/write-doc` command remain the engine that produces one doc through `ground → GATE 1 (scope) → capture → draft → cross-model review → revise → GATE 2 (result)`. This design adds a **doc-type mode** (`user-guide` | `maintainer` | `agents-md`) that selects which investigator, coverage critic, scope rulebook, and leak model the engine dispatches. This modifies the shipped engine rather than adding a passive default: `documentation-pipeline/SKILL.md`, `commands/write-doc.md`, `agents/doc-writer.md`, `agents/doc-editor.md`, and `agents/doc-reviser.md` all hardcode agent names or the `writing-documentation` reference today, and mode threading replaces those literals with a dispatch table. The `user-guide` mode reproduces current behavior, so a bare `/write-doc` invocation is behavior-preserving.

**Project layer (new).** A `documenting-a-project` skill and `/document-project <repo-path>` command orchestrate a whole project: run the survey, hold the set gate, invoke the per-doc engine once per doc in the right mode, and maintain a ledger. It is pure orchestration and dispatches only bundled agents.

### Which agents the mode governs, and which it does not

The mode selects the investigator, the coverage critic, the **scope rulebook**, and the **leak model**. The scope rulebook governs three agents:

- `doc-writer` drafts against the mode's scope rulebook.
- `doc-editor` reviews against the mode's scope rulebook. Today it reads `writing-documentation`, whose scope section (`writing-documentation/SKILL.md:26-33`) mandates surface-not-machinery and "no implementation detail." Left unchanged, the editor would flag a maintainer doc's architecture prose as a scope violation.
- `doc-reviser` rewrites against the mode's scope rulebook. Same reason: it currently reads `writing-documentation` and would strip machinery content "from intent."

The **leak set** is two parts, and both are mode-dependent:

- The leak *model* — which categories count as sensitive — is defined by the scope rulebook. `writing-documentation` treats implementation detail, internal names, table/column names, and API routes as leaks; `writing-internals` inverts that (only secrets, tokens, credentials, internal hostnames, and real production data are sensitive; architecture and module names are the permitted subject).
- The leak *list* — the repo-specific instances found under that model — is produced by the mode's investigator. In `user-guide` mode `doc-investigator` produces the DO-NOT-LEAK list (`doc-investigator.md:16`); in `maintainer`/`agents-md` mode `doc-internals-investigator` produces the inverted leak list. This is a distinct artifact from the ✅/⚠️/🚫 agent-boundary block: the leak list governs what must not appear in any emitted artifact (doc prose *or* `AGENTS.md`), while the boundary block is agent-action permissions destined for `AGENTS.md`.

Every leak-aware consumer reads **both** the mode's leak model (categories, so it can judge a newly written sentence the investigator never saw) and the mode's leak list (the specific instances). Handing a consumer only the list would degrade its leak check to a string search and lose the categorical judgment the v2.1.0 fact-checker does today (`doc-fact-checker.md:24`, "grep the draft for any implementation detail that should not be there"). The consumers:

- `doc-fact-checker`'s claim-verification (does the draft match source) is mode-neutral and shared as-is. Its leak check is not: today it greps for the user-guide DO-NOT-LEAK list (`doc-fact-checker.md:17,24`), which under `maintainer` mode would flag the permitted subject. It reads the mode's leak model and list. This is the agent AC3.4 relies on.
- `doc-writer` (`doc-writer.md:40`), `doc-editor` (`doc-editor.md:20`, "implementation leak"), and `doc-reviser` (`doc-reviser.md:39`) each hardcode the user-guide leak framing in their prompt bodies. Their prompts must defer the leak definition to the mode's leak model and list rather than naming DO-NOT-LEAK / implementation-leak literally, or a maintainer-mode run treats its own subject as a violation.
- `doc-agents-md` emits a committed file, so it is a leak consumer too: it must not reproduce a leak-list instance in `AGENTS.md`, nor derive one into a boundary line (a ✅/⚠️/🚫 rule that names a real internal hostname is still a leak). A committed `AGENTS.md` is a broader leak surface than a doc page, since coworkers pull the whole repo.

Genuinely mode-agnostic: `doc-screenshooter` (only ever in scope for `user-guide`).

### The mental model that resolves the doc dependencies

The **internals investigation** is the unit `AGENTS.md` depends on; the **maintainer doc** is an optional second consumer of that same investigation. Concretely:

- `AGENTS.md` alone → run the full `doc-internals-investigator`, GATE 1, distill, result gate.
- `AGENTS.md` + maintainer doc → one investigation feeds both.
- maintainer doc alone → investigation, full doc pipeline, its two gates.

The investigator always runs at full depth; there is no shallow variant. The reason is a real lower bound: a defensible ✅/⚠️/🚫 boundary block needs to know module boundaries, data flow, and invariants, because "never touch X" is only defensible once you know what depends on X. So even the `AGENTS.md`-alone case needs a genuine investigation, not a template. A maintainer doc needs that same understanding as its floor. It may need marginally more digging to source its why-decisions (rejected alternatives, historical coupling), but that delta is authored narrative plus targeted follow-up, not a different investigation depth. Rather than maintain two grounding paths, one investigator produces a superset artifact and each doc writes up the sections it needs. The consequence, stated plainly: `AGENTS.md`-alone is not a cheap path, and a lighter variant is deferred to a future version alongside the process/runbook modes.

### Mode dispatch

The engine selects components by mode:

| Mode | Investigator | Coverage critic | Scope rulebook (writer/editor/reviser) | Leak model + list (all leak-aware agents) |
|------|-------------|-----------------|----------------------------------------|-------------------------------------------|
| `user-guide` (default) | `doc-investigator` | `doc-coverage-critic` | `writing-documentation` | user-guide model + DO-NOT-LEAK list |
| `maintainer` | `doc-internals-investigator` | `doc-internals-critic` | `writing-internals` | inverted model + inverted leak list |
| `agents-md` | `doc-internals-investigator` | (none; signal-density check at gate) | n/a — no writer/editor/reviser; `doc-agents-md` distills | inverted model + inverted leak list |

The leak-aware agents are `doc-writer`, `doc-editor`, `doc-reviser`, `doc-fact-checker` (prose docs), and `doc-agents-md` (the committed `AGENTS.md`). The leak *model* comes from the mode's scope rulebook (`writing-internals` supplies it for both `maintainer` and `agents-md`); the leak *list* comes from the mode's investigator. `doc-screenshooter` is mode-agnostic and user-guide-only. Mode reaches the engine as an explicit argument the orchestrator passes on dispatch; `/write-doc` gains an optional trailing mode argument defaulting to `user-guide`.

### Data flow (default 3-doc run)

```
/document-project <repo>
  → doc-surveyor: classify repo, inventory existing docs, extract set-gate
                  facts bundle, ask safe-capture questions
  → SET GATE (approve doc set + per-doc write-vs-audit + scope)
  → project orchestrator picks a run-scoped scratch dir and threads its path
    into each per-doc engine dispatch below
  → per doc, sequenced:
      user guide (mode=user-guide): full engine, GATE 1 + GATE 2
      maintainer  (mode=maintainer): doc-internals-investigator writes the
                  internals grounding artifact into the threaded scratch dir;
                  GATE 1 + GATE 2
      AGENTS.md   (mode=agents-md): reuse the internals grounding artifact +
                  set-gate-corrected facts; doc-agents-md distills;
                  single result gate
  → update the committed ledger in <repo>/docs/
```

### The internals grounding artifact (contract)

The internals investigation writes a **run-scoped artifact** so the maintainer writer and the `AGENTS.md` distiller both consume one investigation across separate engine invocations within a single orchestrator run.

- **Location:** a scratch directory the orchestrator chooses once per run and threads as an explicit argument into every per-doc engine dispatch, the same way the screenshot path passes an explicit `--out-dir`. It is *not* written into the target repo: writing there would risk committing an internals dump on a crash and would need to mutate the target's `.gitignore`, which the plugin has no authority to do. For a standalone `/write-doc mode=maintainer` or `mode=agents-md` run (no project orchestrator), the per-doc engine picks the scratch path itself, since a single invocation holds it for its own lifetime.
- **Net-new machinery, not inherited:** the run-id keying, the concurrent-run isolation, and the on-entry/on-exit cleanup below are new work. The only pattern reused from the screenshot path is the one rule "write raw output to a gitignored scratch dir, promote only approved artifacts." The scratch root is orchestrator-chosen (a temp dir it controls), not a fixed plugin-install location, because the plugin install root is shared and read-mostly.
- **Run isolation:** the orchestrator keys the scratch dir per run, so concurrent `/document-project` runs on the same repo do not collide on the artifact.
- **Schema (pinned):** machinery map (data flow, module boundaries), invariants, why-decisions, extracted build/test/run facts, an **inverted leak list** (repo-specific secrets/tokens/hostnames/real-data instances that must stay out of any emitted artifact — doc prose or `AGENTS.md`), and an explicit **agent-boundary block** structured as ✅ always / ⚠️ ask-first / 🚫 never — the exact structure `doc-agents-md` distills from. The leak list and the boundary block are separate fields: the first governs doc prose, the second governs agent actions. The investigator and both consumers share this pinned schema; a schema-validity check (AC3.1) guards against drift.
- **Lifetime:** run-scoped. The orchestrator cleans the run's scratch dir on entry (idempotent, so a prior crash leaves no residue) and on exit. Reuse across engine invocations is same-run only; a resumed run whose scratch artifact is gone re-investigates rather than reading a stale file. The durable, committed record is each doc's **page-to-source map**, so a later audit re-grounds against current source.

### Ask, don't guess (pipeline norm)

Agents stop and ask the runner rather than guess whenever a decision affects leak risk or ground truth is genuinely ambiguous. "What is safe to screencap" is an explicit unknown the pipeline must ask about, never infer. This lands as concrete instructions in `doc-surveyor` (asks which data/accounts/verticals/fixtures are safe and whether a safe mode exists) and the mode agents, not as a slogan. `capture.mjs`'s code-enforced safe-mode assertion remains the backstop.

## Existing Patterns

Investigation of `plugins/kms-docs` (v2.1.0) established the patterns this design follows:

- **One-purpose bundled agents** in `agents/*.md`, each with a pinned model and a fixed toolset. New agents (`doc-surveyor`, `doc-internals-investigator`, `doc-internals-critic`, `doc-agents-md`) follow this shape and pin models.
- **Orchestrator-skill pattern:** `documentation-pipeline/SKILL.md` dispatches agents and holds gates. `documenting-a-project` is a second orchestrator that composes the first, matching the existing shape rather than inventing a new one.
- **Standalone posture (verified clean):** no ed3d dependency, no *required* `kms-human-voice` (its rules are restated inline in `writing-documentation`, per `writing-documentation/SKILL.md:8`), no general-purpose-agent fallback (`documentation-pipeline/SKILL.md:26` states this explicitly). New pieces preserve this.
- **Scratch-dir + promote workflow:** drafts and screenshots live in a gitignored scratch dir the orchestrator picks, and are promoted on approval. The grounding artifact follows the same promote rule; its run-id keying and cleanup lifecycle are new (see the artifact contract).
- **Page-to-source map** as the durable doc↔code link, and a **code-to-docs backlink note** in the nearest `AGENTS.md`/`CLAUDE.md`. Retained; the plugin still only *appends a backlink* to those files and does not otherwise author `CLAUDE.md`.
- **Describe-and-link, never duplicate** (`writing-documentation/SKILL.md:24`). This design applies it to skills too: the shared prose-voice rules move to one in-plugin fragment both scope rulebooks reference, rather than duplicating the tells checklist across `writing-documentation` and `writing-internals`.

Divergence: the per-doc engine gains mode-branching it did not have. Justified because the alternative (a separate parallel pipeline per doc type) duplicates the review machinery and the two-gate quality story.

### Phase 1: Mode-parameterize the per-doc engine

### Phase 5: `AGENTS.md` distillation flow
**Goal:** A gated `AGENTS.md` distilled from internals grounding + extracted facts.

**Components:**
- `plugins/kms-docs/agents/doc-agents-md.md` — consumes the grounding artifact (machinery map, ✅/⚠️/🚫 boundaries, inverted leak list, and the mode's leak model) and its extracted facts; assembles conventional sections with commands taken verbatim from the facts; distills the boundaries; inserts `[TODO]` placeholders for human judgment. Applies two signal-density filters (below). **Respects the leak model and list:** it must not reproduce a leak-list instance in `AGENTS.md` nor emit a boundary line that names one (a ✅/⚠️/🚫 rule quoting a real internal hostname is a leak in a committed file). Model pinned (opus, for the judgment). No separate editor/reviser pass; the single result gate is the review.
  - *Mechanical filter:* reject any line equal to a manifest field value after normalizing whitespace and case, or within a small fixed edit distance. Deterministic string check.
  - *Judgment filter:* reject advice that would hold for an arbitrary repo (generic filler). Opus judgment, backed by the result gate, not a mechanical check.
- Facts source: the investigation's extracted facts. Under `/document-project`, set-gate corrections override those facts on conflict (`docsuite.AC5.5`).
- Single result gate: drafted file, extracted commands (unverified ones flagged), `[TODO]`s, and the optional `CLAUDE.md` bridge.

### Phase 7: Project orchestration
**Goal:** `/document-project` runs survey → set gate → per-doc engine, with tracking and resume.

**Components:**
- `plugins/kms-docs/skills/documenting-a-project/SKILL.md` and `plugins/kms-docs/commands/document-project.md`.
- **Scratch-dir lifecycle (owned here):** the orchestrator picks a run-id-keyed scratch dir, cleans it on entry (idempotent) and on exit, and threads its path into every per-doc engine dispatch. This is the single owner of the grounding-artifact lifecycle.
- **Set gate:** approve/adjust/reject the doc set, per-doc write-vs-audit, and scope. Corrections here are written into the facts record the investigation and `AGENTS.md` consume, overriding investigation-derived facts on conflict. A wholesale rejection aborts cleanly with no ledger written.
- **Ledger:** `<repo>/docs/.docsuite-ledger.md` (or `.json`), a **committed** per-project record with per-doc status. States: `pending | grounded | investigation-done | gate1-passed | drafted | gate2-passed | done | doc-failed | investigation-failed`. `grounded` marks the user-guide grounding milestone; `investigation-done` marks the maintainer/agents-md investigation milestone (the same pipeline stage under each mode's name). Read on entry; **skip `done` docs on re-invocation** (resume). On resume, a doc marked `investigation-done` whose scratch artifact is gone is demoted and re-investigated, not skipped. Owned solely by `/document-project`; direct `/write-doc` never touches it (`docsuite.AC5.6`). A `doc-failed` or `investigation-failed` doc can be reset to `pending` by re-approving it at the set gate on the next run.
- **Failure policy:** an `investigation-failed` maintainer doc halts the `AGENTS.md` that depends on its grounding; a `doc-failed` maintainer doc (valid investigation, failed at drafting/GATE 2) does not block `AGENTS.md`. Independent docs continue in both cases.
- **Concurrent-run guard:** a lock file prevents a second `/document-project` run on the same repo from starting while one holds the lock. Combined with the run-id-keyed scratch dir, runs never share state.
- **Sequencing:** survey → user guide → maintainer → `AGENTS.md`.
- **Audit vs write** passed to the engine as explicit execution intent (audit re-grounds and revises an existing page; write starts fresh).

**Dependencies:** Phases 1, 4, 5, 6.

**Done when:** A full 3-doc run hits exactly the 6 gates, tracks status in the committed ledger, resumes without re-running `done` docs (re-investigating when the scratch artifact is gone), blocks `AGENTS.md` on a failed maintainer investigation but not on a failed maintainer draft, and aborts cleanly on set-gate rejection (`docsuite.AC5.1`–`docsuite.AC5.6`).

## Additional Considerations

**Extensibility (the deferred `process`/`runbook` pivot).** The mode-dispatch table (Phase 1) and the surveyor's taxonomy (Phase 6) are the two extension points. Adding a future `runbook` or `process` mode means one row in the dispatch table + one taxonomy entry + its agents/rulebook, with no change to the orchestration or gate machinery. Noted so the pivot stays cheap; not built in v1.

**Skill count / phase scope.** Eight phases, at the writing-plans limit. Facts extraction must live in the investigator (Phase 3), not the surveyor, because `AGENTS.md` (Phase 5) consumes those facts and must work without a surveyor. Given that constraint, Phases 1–5 form a self-contained first plan (engine + maintainer + `AGENTS.md`, closed because the investigator supplies facts), and Phases 6–8 (survey + orchestration + release) a second. Any split must keep Phases 1–5 together.

**Monorepos.** The pinned agent-boundary block is a single flat ✅/⚠️/🚫 structure, which assumes one set of boundaries for the repo. A monorepo with different rules per package does not fit. v1 has the surveyor flag such repos at the set gate and treats per-package boundary blocks as out of scope, rather than producing a boundary block that is wrong for half the repo.

**AGENTS.md quality risk.** Research showed naive auto-generated `AGENTS.md` files underperform. The mitigation is structural: content comes only from real investigation (never a template), manifest-derivable lines are dropped by a deterministic string check, generic advice is rejected by opus judgment plus the result gate, and the file is presented at its gate as human-curated scaffolding, not a finished artifact. The two filters differ in kind: one is mechanical and testable with a fixed oracle, the other is model judgment tested against a labeled fixture.
